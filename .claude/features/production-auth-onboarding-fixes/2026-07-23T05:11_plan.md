# Production bugfixes: onboarding lockout, auth recovery, friendship re-add

Three production issues, diagnosed to root cause. Two share one antipattern:
**an empty/unknown runtime state is trusted as authoritative** — an empty local
membership cache is read as "user has no band," and a hung network call leaves
no escape. The plan leads with the test coverage that would have caught each,
per the brief.

---

## Issue A — Existing band members forced into new-user onboarding (CRITICAL)

**Symptom:** Pre-update members log in, land on the "join solo / join a band"
get-started screen, and entering their band code errors "already a member." The
whole band is locked out.

**Root cause:** `AuthContext.loadUserData` (`src/contexts/AuthContext.tsx:481-571`)
resolves band membership from **local IndexedDB only** (`db.bandMemberships`,
:495-499). Commit `3e01fe4` added a guard that only honors the persisted
`currentBandId` if it appears in a set derived from that local read
(`validBandIds.has(bandId)`, :552), and `applyPersonal()` (:538-543)
**deletes `currentBandId` from localStorage** on a miss.

Pre-update users carry stale localStorage (`currentBandId` + `last_full_sync`)
but an **empty** local membership store — `SyncEngine.performInitialSync` is
skipped because `last_full_sync` is present, and no incremental sync refills
`band_memberships`. So the guard sees an empty set → downgrades a valid member
to Personal → wipes their band → routes them to onboarding. The join-code path
(`BandMembershipService.joinBandWithCode`) reads membership **cloud-first**,
correctly finds the membership, and returns "already a member" — a dead end.
The two code paths disagree on the source of truth.

**Fix (this branch):** Make `loadUserData` resilient. When the local membership
read is empty, fall back to `repository.getUserMemberships(userId)` (cloud-first,
and it back-fills IndexedDB — same source `switchBand`/`joinBandWithCode` trust).
Resolve each band via `repository.getBand` when it's missing locally. Never
destroy the persisted `currentBandId` on an _unresolved_ (offline + empty) read —
only clear it on an authoritative "personal" intent or an authoritatively-empty
membership list. Net: an existing member is never downgraded to Personal because
of a cold cache.

## Issue B — Auth bypass + unrecoverable empty dashboard (CRITICAL)

**Symptom (B1):** Pages reachable unauthenticated. **Symptom (B2):** login times
out; refresh lands on an empty dashboard (no user/band); logout does nothing;
only a cache purge recovers.

**Root cause:** The route guard's notion of "authenticated" is derived entirely
from two localStorage mirrors (`currentUserId` + `rock_on_session`) and is
decoupled from both the real Supabase session and `AuthContext` hydration
(`src/hooks/useAuthCheck.ts:105-179`).

- **B1 (bypass):** the session monitor (`AuthContext.tsx:123-142`) re-reads only
  the localStorage mirror, never the real session, so a revoked-but-not-expired
  token keeps rendering protected content until the ~1h token clock passes.
- **B2 (stuck empty):** the guard passes on the localStorage keys while
  `loadInitialSession`'s sync fails/times out (`:242-249`), so `loadUserData`
  runs against an empty IndexedDB → `currentUser=null`, `currentBand=null` →
  empty dashboard. Refresh repeats it. **Nothing transitions the user out**
  because the guard only redirects when the keys are _absent_.
- **No escape:** `AuthContext.signOut` (`:630-649`) `await`s a network
  `supabase.auth.signOut()` with **no timeout and no `finally`**, and the
  guaranteed local `logout()` is gated behind it. On the same degraded network
  that caused B2, signOut hangs → the button does nothing → cache purge is the
  only recovery.

**Fix — split into two phases by risk:**

**Phase 1 (this branch, safe, high-value):** Guarantee the escape hatch.
Rework `signOut` to clear local session state **first and unconditionally**,
then attempt the network signOut under a bounded timeout that can never block.
This alone removes the "must purge cache" state: from any broken screen the user
can sign out and back in. (Local `logout()` already dispatches `auth-logout`,
which the guard reacts to → redirect to `/auth`.)

**Phase 2 (proposed, needs review — NOT in this PR):** Make the guard
fail-closed against real state:

1. `AuthContext` exposes an authoritative `authStatus: 'checking' |
'authenticated' | 'unauthenticated'` that resolves only after
   `loadInitialSession` confirms a real session; the guard renders protected
   content only on `'authenticated'`.
2. The periodic session monitor verifies the **real** session (network truth,
   e.g. `getUser()`), only expiring on a definitive null (never on a transient
   offline error), to close B1 (revoked tokens).
3. A dedicated recovery screen for the authenticated-but-unhydrated state
   (`currentUser` null after load) with **Retry sync** + always-visible
   **Sign out** — instead of a silent empty dashboard.

Phase 2 changes the guard's core semantics and could log users out on
transient offline errors if done carelessly — it wants human review before
shipping, so it is documented here and deferred, not merged blind.

## Issue C — Friend request blocked after unfriend

**Symptom:** After unfriending, re-adding says "you're already friends."
(Decline/cancel already re-add fine — fixed earlier in `e14cbbd`.)

**Root cause:** Accepting a request leaves a permanent `friend_requests` row at
`status='accepted'` (the accept trigger writes `friendships` but keeps the
request row; `friends.sql:110-129`), occupying the `UNIQUE(requester,addressee)`
slot (`friends.sql:59`). `FriendService.unfriend` (`src/services/FriendService.ts:396-404`)
deletes only the `friendships` row — nothing clears the stale `accepted` request
row. On re-add, `insertRequest` hits the unique slot and only reactivates
`declined`/`cancelled` rows (`:346`), so the `accepted` row falls through to
"You're already friends" (`:355-361`).

**Fix (this branch):**

1. DB: `AFTER DELETE ON public.friendships` SECURITY DEFINER trigger that
   deletes any `friend_requests` row between the two users (either direction).
   Covers all callers and future-proofs cleanup.
2. App self-heal: in `insertRequest`, when the conflicting row is `accepted`
   but `are_friends(me, target)` is false (they already unfriended), reactivate
   it to `pending` instead of erroring. Repairs orphaned rows already in prod.

## Test strategy (LEAD — build these first)

### Unit

- **`AuthContext.loadUserData` (new `tests/unit/contexts/AuthContext.test.tsx`)** —
  the regression is here and is currently untested.
  - Empty local `bandMemberships` + stored `currentBandId` + cloud membership
    present ⇒ `currentBandId` preserved, `hasBand` true, NOT routed to personal.
  - `rockon.context='personal'` ⇒ personal context honored.
  - Valid local band ⇒ band selected (precedence table pinned).
  - Cloud fallback throws (offline) + stored `currentBandId` ⇒ id preserved, not wiped.
- **`AuthContext.signOut`** — when `authService.signOut()` rejects AND when it
  hangs (never resolves) ⇒ local state + localStorage cleared and `auth-logout`
  dispatched within the bounded timeout.
- **`FriendService.insertRequest`** —
  - conflicting row `accepted` + `are_friends` false ⇒ reactivates to pending, `ok:true`.
  - conflicting row `accepted` + `are_friends` true ⇒ "You're already friends".
  - (regression lock) `declined`/`cancelled` still reactivate.

### E2E (Playwright)

- **`tests/e2e/auth/existing-member-onboarding.spec.ts`** — seed a server-side
  membership, clear IndexedDB, keep `currentUserId`/`currentBandId`/`last_full_sync`,
  log in ⇒ lands on the app (not `/get-started`). Reproduces the production lockout.
- **`tests/e2e/auth/signout-recovery.spec.ts`** — in a degraded state (network
  stubbed to hang) the Sign Out control clears the session and lands on `/auth`
  within a bounded time, on both mobile and desktop viewports.
- **`tests/e2e/friends/re-add-after-unfriend.spec.ts`** — A→B, B accepts, A
  unfriends, A re-adds B ⇒ succeeds (no "already friends" toast). Plus regression
  legs for decline→re-add and cancel→re-add.

### Phase-2 tests (ship with phase 2)

- Guard state-machine: keys present + valid mirror but real session null ⇒
  unauthenticated/redirect (B1).
- No flash of protected content on hard refresh (spinner until resolved).
- Recovery screen appears for authenticated-but-empty hydration.
