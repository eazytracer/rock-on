# Research: Social Catalog

## Feature Request Summary

A unified initiative covering three tightly coupled capabilities:

1. **Personal Song Catalog** — Users maintain a personal song library independent of any band (songs with `contextType='personal'`). Schema already supports this; UI does not yet exist.

2. **Personal Setlists** — Setlists that belong to a user rather than a band. Requires relaxing `setlists.band_id` from `NOT NULL` and adding `context_type`/`context_id` fields.

3. **Jam Sessions** — Users share their personal song catalog with other users via a QR code or short hash to find **songs they have in common** for an impromptu jam. One user hosts, others join. The resulting matched-song list can optionally be saved as a personal setlist. Free users get 1 active session; pro users get unlimited.
   - **Anonymous read-only access**: Non-authenticated users can view a jam session's matched songs via a scoped `view_token` and a Supabase Edge Function — no login required, serving as an in-app "try before you sign up" experience.

These three features are developed as a single package because each depends on the one before it.

## Technical Context

### Existing Patterns to Follow

**Invite Code Pattern** (`invite_codes`, line 134 of migration):
The jam session `short_code` reuses this exact pattern — a short human-readable unique code, expiry, max uses, `is_active` flag. The `jam_sessions` table owns the `short_code` directly (no separate invites table needed for v1).

**Setlist Fork Pattern** (`setlists.forked_from`, `setlists.source_setlist_id`):
When a jam session is saved as a personal setlist, `setlists.jam_session_id` records provenance, and `setlists.tags` includes `'jam'` to allow filtering.

**Song Context Pattern** (`songs.context_type`, `songs.context_id`):
Personal songs already use `contextType='personal'`, `contextId=userId`. The jam session matcher queries personal songs using this filter — same API, zero new query logic needed.

**SECURITY DEFINER Helper Pattern** (`is_band_member()`, `is_band_admin()`, line 860+):
A new `is_jam_participant(p_session_id, p_user_id)` function follows the identical pattern — SECURITY DEFINER, owned by postgres, prevents RLS recursion.

**Realtime Pattern** (Section 9 of migration):

```sql

ALTER PUBLICATION supabase_realtime ADD TABLE jam_song_matches;
```

`RealtimeManager.ts` already handles Supabase Realtime subscriptions — jam session live updates plug into this.

### Song Matching Algorithm

The central technical challenge. Users may have the same song stored as:

| User A                 | User B                      | Should match?            |
| ---------------------- | --------------------------- | ------------------------ |
| "The Black Parade"     | "Black Parade"              | ✅ Yes                   |
| "The Counting Crows"   | "Counting Crows"            | ✅ Yes                   |
| "Don't Stop Believin'" | "Don't Stop Believing"      | ✅ Yes                   |
| "Bohemian Rhapsody"    | "Bohemian Rhap."            | ⚠️ Fuzzy                 |
| "Sweet Home Alabama"   | "Sweet Home Alabama (Live)" | ⚠️ Fuzzy                 |
| "Hurt" (NIN)           | "Hurt" (Johnny Cash)        | ❌ No — different artist |

**Multi-tier matching strategy:**

**Tier 1 — Exact normalized match (primary):**

```typescript
function normalize(s: string): string {
  return s
    .toLowerCase()
    .replace(/^(the|a|an)\s+/i, '') // Remove leading articles
    .replace(/[''`\u2018\u2019]/g, "'") // Normalize apostrophes
    .replace(/[^a-z0-9 ]/g, '') // Strip non-alphanumeric
    .replace(/\s+/g, ' ') // Collapse whitespace
    .trim()
}
// "The Black Parade" → "black parade"
// "Black Parade"     → "black parade"  ✅ MATCH
// "Don't Stop Believin'" → "dont stop believin" ✅ MATCH
```

Both `normalized_title` AND `normalized_artist` must match (prevents cross-artist false positives).

**Tier 2 — Fuzzy title match with exact artist (flagged):**
Levenshtein distance ≤ 2 (or Jaro-Winkler ≥ 0.92) on `normalized_title`, with exact `normalized_artist` match. Shown in UI as "possible match — confirm?" — host can accept or dismiss.

**Tier 3 — Manual link:**
Host can manually pair two songs that the algorithm missed. Stored as a `match_confidence='manual'` record in `jam_song_matches`.

**Implementation recommendation:**

- Store `normalized_title` and `normalized_artist` as **indexed columns** on the `songs` table, populated by a BEFORE INSERT/UPDATE trigger using a `normalize_text()` SQL function.
- This enables efficient server-side matching with a single indexed query rather than a full table scan + application-layer string comparison.
- The normalization function lives in PostgreSQL so both the app and the Edge Function use the same logic.

```sql
CREATE OR REPLACE FUNCTION normalize_text(input TEXT)
RETURNS TEXT
LANGUAGE plpgsql IMMUTABLE STRICT AS $$
BEGIN
  RETURN regexp_replace(
    regexp_replace(
      regexp_replace(lower(input), '^(the|a|an)\s+', '', 'i'),
      '[^a-z0-9 ]', '', 'g'
    ),
    '\s+', ' ', 'g'
  );
END;
$$;
```

### Unauthenticated Read-Only Access

**Architecture: Supabase Edge Function**

A Supabase Edge Function serves jam session data to unauthenticated users. This approach:

- Requires **zero anon RLS policies** — no tables are opened to unauthenticated requests
- Runs as `service_role` internally, but only ever returns the pre-approved subset of data
- Is easy to rate-limit (standard Edge Function rate limiting)
- Is the cleanest security boundary

**Token flow:**

1. Host creates jam session → server generates `view_token` (UUID, hashed with SHA-256 before storing)
2. `view_token` (pre-hash) is embedded in the shareable URL: `https://rockon.app/jam/view/{short_code}?t={raw_token}`
3. Unauthenticated user visits the URL
4. React renders `JamViewPage` (public route, outside `ProtectedLayoutRoute`)
5. `JamViewPage` calls Edge Function: `GET /api/jam/{short_code}?t={raw_token}`
6. Edge Function hashes the token, queries `jam_sessions` by `short_code + hashed view_token`, checks `status='active'` and `view_token_expires_at > NOW()`
7. Returns only: session name, host display name, matched songs (title/artist only), participant count
8. Never returns: participant user IDs, emails, full song catalogs, band data

**What the unauthenticated view shows:**

- Session name (or "Unnamed Jam Session")
- Host's `display_name` (from `user_profiles`, never email or user ID)
- List of matched songs with title, artist, and match confidence badge
- Total participant count (no names)
- Prominent CTA: **"Join this jam session — Sign up free"** → `/auth?view=signup&redirect=/jam/{short_code}`

**Edge Function location:** `supabase/functions/jam-view/index.ts`

**New public route in `App.tsx`:**

### RLS Policy Strategy

Jam sessions are **not band-scoped** — there is no `band_id` anchor. All existing helpers (`is_band_member`, `is_band_admin`) are irrelevant here. New helper:

```sql
CREATE OR REPLACE FUNCTION is_jam_participant(p_session_id UUID, p_user_id UUID)
RETURNS BOOLEAN
LANGUAGE sql SECURITY DEFINER
SET search_path = public STABLE AS $$
  SELECT EXISTS (
    SELECT 1 FROM public.jam_participants
    WHERE jam_session_id = p_session_id
      AND user_id = p_user_id
      AND status = 'active'
  );
$$;
ALTER FUNCTION is_jam_participant(UUID, UUID) OWNER TO postgres;
GRANT EXECUTE ON FUNCTION is_jam_participant(UUID, UUID) TO authenticated;
```

**RLS policies:**

```sql
-- jam_sessions
-- SELECT: host or active participant can read; also anyone checking by short_code for join flow
CREATE POLICY "jam_sessions_select"
  ON public.jam_sessions FOR SELECT TO authenticated
  USING (
    host_user_id = (select auth.uid()) OR
    is_jam_participant(id, (select auth.uid()))
  );

CREATE POLICY "jam_sessions_insert"
  ON public.jam_sessions FOR INSERT TO authenticated
  WITH CHECK (host_user_id = (select auth.uid()));

CREATE POLICY "jam_sessions_update"
  ON public.jam_sessions FOR UPDATE TO authenticated
  USING (host_user_id = (select auth.uid()));

CREATE POLICY "jam_sessions_delete"
  ON public.jam_sessions FOR DELETE TO authenticated
  USING (host_user_id = (select auth.uid()));

-- jam_participants
CREATE POLICY "jam_participants_select"
  ON public.jam_participants FOR SELECT TO authenticated
  USING (is_jam_participant(jam_session_id, (select auth.uid())));

CREATE POLICY "jam_participants_insert_self"
  ON public.jam_participants FOR INSERT TO authenticated
  WITH CHECK (user_id = (select auth.uid()));

CREATE POLICY "jam_participants_delete_self_or_host"
  ON public.jam_participants FOR DELETE TO authenticated
  USING (
    user_id = (select auth.uid()) OR
    EXISTS (SELECT 1 FROM jam_sessions WHERE id = jam_session_id AND host_user_id = (select auth.uid()))
  );

-- jam_song_matches
CREATE POLICY "jam_song_matches_select"
  ON public.jam_song_matches FOR SELECT TO authenticated
  USING (is_jam_participant(jam_session_id, (select auth.uid())));

-- No INSERT/UPDATE/DELETE from application layer — computed by JamSessionService via service_role
```

> **Note on personal songs RLS:** The current songs policies (`songs_select_band_members_only`) only allow `context_type='band'`. Personal song support requires adding a parallel policy: `context_type='personal' AND context_id::uuid = auth.uid()`. This is a required change as part of Phase 1.

## Risk Analysis

### High Risk

- **Song matching false positives/negatives**
  - **Impact:** Core feature value is lost if users see wrong matches or miss valid songs
  - **Mitigation:** Multi-tier matching with unit test suite of 50+ real-world edge cases. Fuzzy matches shown with distinct visual treatment ("Possible match — is this the same song?"). Host can manually accept or dismiss. Normalization logic lives in SQL so it's consistent across app + Edge Function.

- **Personal songs RLS — must not leak songs between users**
  - **Impact:** User A's personal songs visible to User B during a jam session
  - **Mitigation:** `jam_song_matches` stores only the canonical match + display title/artist — never a direct reference to the raw `songs` row of another user. Participants cannot query each other's `songs` table directly via RLS. The matching computation runs server-side in `JamSessionService` using elevated privileges, writes only to `jam_song_matches`, then drops access.

- **`setlists.band_id` relaxation is a breaking schema change**
  - **Impact:** Any existing query assuming `band_id IS NOT NULL` will need updating
  - **Mitigation:** Pre-1.0 project — safe to update baseline migration directly. Add `context_check` constraint to enforce internal consistency. Audit all setlist queries in `RemoteRepository.ts` and `LocalRepository.ts` for `band_id` assumptions.

- **Tier gating depends on `account_tier` column not yet on `users` table**
  - **Impact:** Cannot enforce session limits without the column
  - **Mitigation:** Add the column (DEFAULT 'free') as part of Phase 3 even if enforcement is a stub. Zero behavioral change for existing users.

### Medium Risk

- **Unauthenticated view — token security**
  - **Impact:** If `view_token` is guessable or reused, private jam session data leaks
  - **Mitigation:** Token is a crypto.randomUUID() (122 bits of entropy), hashed with SHA-256 before storage. Edge Function checks expiry. Rate limit the Edge Function (5 req/min per IP). Token rotates if host regenerates the invite link.

- **Realtime subscription for non-band tables**
  - **Impact:** `RealtimeManager` currently subscribes using band-scoped channels; jam sessions have no `band_id`
  - **Mitigation:** Jam session subscriptions use a new channel pattern: `jam:session:{session_id}`. May require extending `RealtimeManager` to support non-band channels.

- **Personal song catalog and personal setlists are new UI paradigms**
  - **Impact:** Users may be confused about personal vs. band context
  - **Mitigation:** Clear tab/section labels. Personal items never appear in band views. Consistent visual distinction (color, icon, label).

### Low Risk

- QR code library integration (well-solved, `qrcode.react` is mature)
- Short code collision probability (6 alphanumeric chars = 2.2B possibilities; negligible at app scale)
- Edge Function deployment (Supabase CLI supports this natively)

## Recommended Approach

**Technical strategy:**

- Reuse the `invite_codes` pattern for `short_code` generation — already proven
- Store `normalized_title`/`normalized_artist` as SQL generated columns — consistent, indexed, no app-layer maintenance
- Use Supabase Edge Function for unauthenticated access — cleanest security boundary
- `jam_song_matches` as a pre-computed materialized result table — avoids real-time heavy computation, makes the unauthenticated view trivially fast
- Keep jam sessions entirely separate from the `bands` table — no temporary band records

**Rationale:**

- Personal songs schema already exists (contextType/contextId) — Phase 1 is mostly UI work
- The matching problem is fundamentally a normalization problem, not a search problem — simple SQL function solves 95% of cases
- Edge Function for anon access avoids the security complexity of anon RLS policies on core tables
