# Mobile Redesign Port — Research

## 1. The core reframe (most important finding)

**The real app is already dark-only.** There is no light theme, no `dark:` variants, no
theme toggle. What the prototype calls a "dark refresh" is, in the real app, largely a
**token-consolidation + gap-fill** exercise, not a visual overhaul. This matches the
user's expectation that "the vast majority of the changes are UI-only."

Two concrete problems the current app has that the port fixes:

1. **Two competing oranges are already live.** Legacy `#FE4401` (`energy-orange`, used in
   auth flows — `LoginForm.tsx`, `SignupForm.tsx`, `BandCreationForm.tsx`,
   `ProtectedLayoutRoute.tsx`) vs. newer `#f17827ff` (notes/songs/setlists —
   `SongListItem.tsx:189`, `InlineStatusBadge.tsx:162`, `MarkdownField.tsx`). Neither is
   the prototype's `#ff7a1a`. The port unifies all three to a single `--accent`.
2. **Zero CSS custom properties exist** (`grep var(-- src/index.css` = 0). The dark UI is
   painted with ~65 files of hard-coded arbitrary hex (`bg-[#0a0a0a]`, `border-[#2a2a2a]`,
   `text-[#a0a0a0]`, …). A re-skin today means editing 65 files; after the port it's a
   one-file variable edit.

**Implication for scope:** the biggest UI-wide win is establishing the token system first,
then everything else (existing screens _and_ net-new screens) renders consistently on top
of it. Genuinely net-new screen work is smaller than it looks — most primitives already
exist (see §5).

## 2. Design token system (values to port, from the prototype)

Source: prototype `ui.jsx` + `data.js` (read directly). Establish these as CSS custom
properties in `:root` (`src/index.css`) and mirror into `tailwind.config.js`.

**Surfaces / ink / accent:**

```
--bg-0 #0a0a0a  --bg-1 #121212  --bg-2 #1a1a1a  --bg-3 #1f1f1f  --bg-4 #252525
--border-1 #2a2a2a  --border-2 #3a3a3a
--ink-1 #fff  --ink-2 #c8c8c8  --ink-3 #a0a0a0  --ink-4 #707070  --ink-5 #505050  --ink-6 #3a3a3a
--accent #ff7a1a   (+ --accent-hot / -deep / -soft / -line derived via color-mix)
```

**Semantic:** `--info #5aa9ff  --success #3ec986  --warn #ffb648  --danger #ff5d5d` (each
with a `-soft` bg). Badge tone map: info/success/warn/danger/neutral/accent
(`BADGE_TONE` in `ui.jsx`).

**Type:** Geist (UI) + JetBrains Mono (stats/tempo/keys/IDs/eyebrows). Currently Inter.

**Signature — tuning-warmth spine** (`data.js` `TUNING`): every tuning → a warmth color
(bright/warm = higher pitch, deep/cool = lower): `standard #ffd166` … `open-d #ff7a1a` …
`drop-b #6c3aa6`, `custom #6b7280`. **⚠ Reconcile, do not duplicate:** the real app
already has a tuning color system — `src/utils/tunings.ts` (`CANONICAL_TUNINGS` "Palette A",
`tuningColor`, `tuningLabel`, `canonicalTuningId`) — plus draft palettes in
`src/pages/DevUIPreview/tuningColors.ts`. Decide which palette wins; merge into one module.

**Avatars:** deterministic 12-color hash palette (`data.js` `AV[]` + `avatarColor`). Real
app already has `src/utils/songAvatar.ts` + `SongAvatar.tsx` — generalize, don't rebuild.

**Instrument spine** (`data.js` `INSTRUMENT`): vox `#ff7a1a`, bvox `#ffb648`, gtr `#5aa9ff`,
bass `#9d6cff`, keys `#3ec986`, drums `#ff5d5d`. (Used by casting/roles.)

**Icons — REUSE WIN:** the prototype hand-rolled a lucide-style icon set (`RK_ICON_PATHS`)
only because it had no bundler. The real app already imports `lucide-react` everywhere. **Do
not port the custom icon system** — map prototype icon names to lucide components.

## 3. Theming architecture — recommendation

**Hybrid, CSS-variables-as-source-of-truth:**

1. Define all tokens as CSS custom properties in `:root` in `src/index.css`.
2. Map them into `tailwind.config.js` (`colors: { 'bg-0': 'var(--bg-0)', accent: 'var(--accent)', … }`)
   so authoring stays in Tailwind classes (`bg-bg-0`, `text-ink-1`, `border-accent`) — the
   codebase's existing idiom — instead of arbitrary `bg-[#…]` literals.
3. Provide a small **TS token module** (mirrors the CSS vars) for the JS-computed colors
   (tuning/avatar/instrument hex used in inline `style={{}}`) so there's one source, not two.
4. Migrate the ~65 arbitrary-hex files to semantic classes incrementally; collapse the two
   oranges to `--accent`.

Why: least code, one-file re-skin, preserves current authoring style, kills the duplication.
Rejected: pure Tailwind-extend (can't cleanly express a swappable ramp, leaves literals);
pure arbitrary-values (keeps the duplication).

## 6. Parity risk register (the regression safety net)

The redesign changed/removed some things; the real app also has depth the design docs
under-represent. Verify each survives the port. **Highest priority (High severity):**

- **Per-item SyncIcon — 5 states** (synced/syncing/pending/error/**unread-with-blue-badge**),
  surfaced on song/setlist/show/practice rows (`SyncIcon.tsx:23`, `useItemSyncStatus.tsx:135`).
  The "icon-only on mobile" change is exactly where this could silently vanish. Also
  `SyncStatusIndicator` (connection dot, pending/failed counts, last-sync time) and
  `OfflineIndicator`, `ConflictResolutionModal`.
- **Song Notes = dual band + personal fields** (`SongNotesModal.tsx`), plus `SongNotesPanel`
  band/personal/**log** tabs and `NoteEntryForm` visibility toggle. Appears in Songs,
  SetlistView, PracticeView.
- **EditSongModal depth:** Spotify search→autofill, Circle-of-Fifths key picker, tuning
  select, reference-links editor (7 types, URL sanitization/XSS guard, auto-detect,
  https-normalize), "also save to personal catalog" linked mirror, unsaved-changes guard.
- **Songs:** Band vs Personal (My Songs) tabs; **Copy to My Songs** + linked-catalog chain
  indicator; filters (tuning/upcoming-show/tags) + 7 sort modes; next-show per song.
- **Setlist item types** song/**section/break** (break has duration+notes, counts to total);
  drag reorder + autosave; per-song notes; show link (forks a per-show copy); Band/Personal
  tabs. Two editors exist (`SetlistViewPage` route + modal in `SetlistsPage`) — don't port one
  and lose the other.
- **Shows:** contacts with **tap-to-call `tel:` / `mailto:`** (add/edit/remove); load-in,
  soundcheck, payment (cents→$), notes; setlist attach forks a per-show copy.
- **Practice Session play-along mode** (`PracticeSessionPage`): 4 responsive layouts
  (TV/tablet-land/tablet-port/mobile) auto+manual w/ localStorage; font S/M/L; **BT
  foot-pedal + keyboard scroll** shortcuts; song stepping; up-next + tuning-change alert.
- **Band admin:** regenerate invite code, **transfer ownership** (type TRANSFER),
  make/remove admin, remove member, permission gating (`canEditBand`/`canInviteMembers`/…),
  edit instruments (+primary star).
- **Jam:** create (seed from personal setlist), join by 6-digit code, **QR share**,
  participants + anon watcher presence, **Common Songs auto-match** (confirm/add/recompute),
  end-session flow (save&end / end-without-saving), anon `/jam/view/:code` 5s-poll public view.

**Medium/low:** show/practice/setlist status badges & filters; markdown notes on

## 7. New features — DB & backend (Events, Friends, Notifications)

All three: **Supabase-only + Realtime**, NOT the band-scoped local-first sync engine
(mirror the jam-tables decision — `JamSession.ts:1`, unified-schema §jam). Each ships as a
**separate incremental migration** (one per feature, idempotent, explicit GRANTs to
`authenticated` AND `service_role`, RLS, SECURITY DEFINER helpers). Recommended sequence:
**Notifications → Friends → Events** (dependency & value order).

### Events — mostly reuse of jam + calendar-events patterns

Reuses: shortCode alphabet (`JamSessionService.ts:31`), display-tuple lineup pattern
(`JamSession.ts:26`), `view_token` unlisted access, `is_jam_participant` RLS helper shape,
calendar-events `scheduled_date/timezone/calendar_uid`. **New:** a distinct `events` table
(do NOT overload `jam_sessions` — different lifecycle), per-event ACL
(`event_participants.access_tier ∈ host|cohost|guest|viewer`), **RSVP** (going/maybe/declined),
lineup **requests** the host approves (`event_lineup_requests`, maps to
`jam_song_matches.is_confirmed`), `visibility ∈ private|unlisted|public`, confirmed
`event_lineup_items` with `source ∈ mine|band|public|external` + `owner_id`. Aggregates into
the Calendar next to shows/practices. Full proposed DDL for all four tables + indexes +
helpers (`is_event_participant`, `event_access_tier`) is in the DB agent findings; carry into
`/plan`.

### Friends — fully net-new (no model exists)

`user_profiles` += `discoverable bool default false` (Discord-style hidden-by-default),
`friend_code` (stable per-user handle, backfill via shortCode gen, then UNIQUE),
`friend_request_policy ∈ everyone|friends_of_friends|code_only`. New tables:
`friend_requests` (directed, `requester≠addressee`, UNIQUE pair) and `friendships`
(**canonical single row**, `user_a < user_b`, avoids dual-write half-friendship bug). Helper
`are_friends(u1,u2)`. QR/code reuses `qrcode.react` + jam shortCode alphabet.

### Notifications — net-new feed

`notifications` (kind ∈ release|activity|event|friend, `read_at` NULL=unread, `payload` JSONB,
`link`); `release_notes` (keyed by version, markdown body) + `users.last_seen_release_version`
gate; digest ("while you were away") computed from `notifications` + existing
`users.last_login` (no extra table). INSERT = service_role only (minted server-side).

### Access-control gotchas (from DB research)

- Unlisted event access → **edge function via `view_token`**, not a broad RLS `USING
visibility='unlisted'` (that leaks all unlisted events — the `jam_sessions_select_by_code`
  over-permissive pattern).
- Friend-request policy (checking the _addressee's_ policy + friends-of-friends) **can't live
  in RLS WITH CHECK** → edge function (`send-friend-request`).
- Friend-code lookup of hidden users → **edge function** (`resolve-friend-code`), not a
  `friend_code=?` RLS policy (still needs row-selectability).
- `friendships` INSERT/DELETE = service_role only (atomic on request-accept).
- Don't re-open global user enumeration — extend the existing `users_select_*` policies
  (baseline `:279-307`) with friend/discoverable policies carefully.

### Sync/repo/realtime impact

New service classes mirroring `JamSessionService`: `EventService`, `FriendService`,
`NotificationService`. RemoteRepository snake↔camel mappers per table. `RealtimeManager`
channels for `event_participants`/`event_lineup_requests` (host console), `friend_requests`
(incoming badge), `notifications` (bell) — each needs `ALTER PUBLICATION supabase_realtime
ADD TABLE` + `REPLICA IDENTITY FULL`. Edge functions per FUNCTIONS.md manifest:
`resolve-friend-code`, `send-friend-request`, `accept-friend-request` (+ optional notification
minting). Update `FUNCTIONS.md` and `lint:migrations` must pass.

## 8. Recommended phasing (small passes)

Ordered to front-load the highest-leverage, lowest-risk work and keep every pass shippable.

- **Pass 0 — Token foundation (UI-only, app-wide):** CSS vars in `index.css` + Tailwind
  mapping + TS token module; unify the two oranges to `--accent`; add Geist/JetBrains Mono.
  Reconcile the tuning palette into one module. No screen restructure. _Biggest consistency win._
- **Pass 1 — Shared primitives:** build the small gaps (static `Badge`, `Eyebrow`,
  bottom-sheet); generalize `Avatar`; confirm `MetaPill`/`Card`/`TuningPill`/`LinkButtons`
  match the new tokens. Establish the reuse baseline before touching screens.
- **Pass 2 — Navigation/shell:** re-skin + wire `BottomNavigation` into `ModernLayout`;
  add Home/Calendar/More routes; reconcile desktop Sidebar. Header bell placeholder.
- **Passes 3–N — Screen re-skins (one area per pass), each guarded by the §6 parity list:**
  Songs → Setlists → Shows → Practices (incl. session) → Band/Settings/Profile → Jam. Migrate
  that area's arbitrary-hex to semantic classes as you go.
- **Pass: Calendar aggregator + Dashboard + More hub** (net-new UI over existing data).
- **Feature: Notifications** (migration + service + bell/center/digest).
- **Feature: Friends** (migration + edge functions + service + friends screens).
- **Feature: Events** (migration + service + host/guest screens + calendar integration).

UI passes (0–N + Calendar/Dashboard/More) need no DB. The three feature blocks are the only
DB work and are independent of the re-skin passes.

## 9. Decisions (defaults adopted; user was away — revisit if needed)

1. **Tuning palette → RESOLVED: `src/utils/tunings.ts` "Palette A" is canonical.** It was
   already approved in the 2026-04-22 UI unification review, and `DevUIPreview/tuningColors.ts`
   states approved palettes move into `tunings.ts` (Palette A already did — they match).
   ⚠ Consequence: the prototype's **"tuning-warmth spine"** (pitch→warmth ordering) is
   **superseded** by Palette A's unique per-tuning identifiable colors. Align the design tokens
   to `tunings.ts`, NOT the reverse. (Example of real app > design docs, as the user flagged.)
2. **Desktop → DEFAULT: sidebar as a desktop superset of the 5-tab IA**; bottom nav on mobile
   only. Least disruption to `ModernLayout`/`ProtectedLayoutRoute`. (A desktop Events surface
   also exists in the prototype for later.)
3. **Event `source` → DEFAULT: derived from the referenced song's ownership/visibility**;
   `external` = no `song_id`, relies on frozen display tuple.
4. **Friendship model → DEFAULT: canonical single row** (`user_a < user_b`).
5. **Notifications → DEFAULT: single `release_notes` row gated by
   `users.last_seen_release_version`** (avoids N×users fan-out); 90-day prune window for feed.
6. **Events on offline calendar → DEFAULT: no. Supabase-only + Realtime** (jam precedent).
   Revisit only if offline event access becomes a hard requirement.
7. **Auth screens → DEFAULT: re-skin LoginForm/SignupForm as part of Pass 0's orange
   unification** (they carry the legacy `#FE4401`).

## 10. Risks

- **65-file hex migration** is broad; do it incrementally per screen pass, not big-bang.
- **Enum drift** (TS union vs SQL CHECK) is runtime-fatal — flagged in open-jam research; keep
  them identical from day one for all new tables.
- **Two setlist editors** — easy to port one and regress the other.
- **SyncIcon on mobile** — the single most likely silent regression.
- **Casting service/tables** must survive the UI shelving.
