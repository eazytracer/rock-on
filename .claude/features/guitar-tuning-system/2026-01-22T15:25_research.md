# Research: Enhanced Guitar Tuning System with Per-String Storage

## Feature Request Summary

Create a sophisticated guitar tuning system that enables:

1. **Per-string note storage** - Store the actual note for each string, not just a label
2. **Variable string count** - Support 4-string bass through 8-string guitars
3. **Retuning effort visualization** - Highlight setlist transitions requiring retuning:
   - Standard → Drop D = 1 string change (quick tune)
   - Standard → Half-step down = 6 strings (full retune or swap)
4. **Pre-built tunings library** - Ship with common tunings already defined
5. **Custom tuning creation** - Allow users to define band-specific tunings

**Primary use case:** When viewing a setlist, musicians need to see where guitar/bass changes are required and how much retuning effort is involved between songs.

## Problem Statement

### Current Limitation

The existing system stores guitar tuning as a simple TEXT field:

```sql
-- Current schema
CREATE TABLE songs (
  ...
  guitar_tuning TEXT,
  ...
);
```

This limits functionality to:

- Display a label ("Drop D", "Standard")
- Cannot calculate retuning effort between songs
- Cannot validate if tuning is possible on user's instrument
- Cannot provide intelligent suggestions for setlist ordering
- No per-string information for detailed analysis

### Real-World Scenario

**Setlist Example:**

1. "Highway to Hell" - Standard (E A D G B E)
2. "Ain't Talkin' 'Bout Love" - Drop D (D A D G B E) → **1 string change**
3. "Everlong" - Drop D (D A D G B E) → **No change**
4. "Killing in the Name" - Drop D (D A D G B E) → **No change**
5. "Black Hole Sun" - Half-step down (Eb Ab Db Gb Bb Eb) → **6 string changes**

**Musicians need to see:**

- Song 1→2: Quick retune (1 string)
- Songs 2→3→4: No change needed
- Song 4→5: Full retune or guitar swap required

## Data Model Design

### Recommended: JSONB Column in Songs Table

Advantages:

- Simple schema - no joins required
- Self-contained tuning data with each song
- Easy to query and filter
- PostgreSQL JSONB has excellent indexing and query support

```typescript
// Type Definition
interface GuitarTuning {
  name: string;           // "Drop D", "Standard", "Custom - MyBand"
  stringCount: 4 | 5 | 6 | 7 | 8;
  strings: string[];      // ["D", "A", "D", "G", "B", "E"] (low to high)
  isCustom: boolean;      // Pre-built vs user-created
  bandId?: string;        // For band-specific custom tunings
}

// Database Schema
ALTER TABLE songs
ADD COLUMN tuning JSONB;

-- Example data
{
  "name": "Drop D",
  "stringCount": 6,
  "strings": ["D", "A", "D", "G", "B", "E"],
  "isCustom": false
}
```

### Why Not a Separate Tunings Table

A normalized approach would require:

```sql
CREATE TABLE tunings (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  string_count INTEGER NOT NULL,
  strings JSONB NOT NULL,  -- Still need JSONB for variable array
  is_custom BOOLEAN DEFAULT false,
  band_id TEXT REFERENCES bands(id),
  created_date TIMESTAMPTZ DEFAULT NOW()
);

ALTER TABLE songs
ADD COLUMN tuning_id TEXT REFERENCES tunings(id);
```

Disadvantages:

- Additional join on every song query
- Orphaned tuning records if songs are deleted
- More complex migration path
- Still needs JSONB for strings array

**Recommendation: JSONB in songs table**

## Retuning Effort Algorithm

```typescript
interface RetuningAnalysis {
  stringsChanged: number // Count of different strings
  effort: 'none' | 'quick' | 'moderate' | 'full'
  details: StringChange[] // Per-string differences
  recommendation?: string // "Consider guitar swap"
}

interface StringChange {
  stringNumber: number // 1-based (1 = low E)
  from: string // "E"
  to: string // "D"
  semitones: number // -2 (E to D)
}

function calculateRetuningEffort(
  fromTuning: GuitarTuning,
  toTuning: GuitarTuning
): RetuningAnalysis {
  // Handle different string counts
  if (fromTuning.stringCount !== toTuning.stringCount) {
    return {
      stringsChanged: -1,
      effort: 'full',
      details: [],
      recommendation: 'Different instrument required',
    }
  }

  // Compare each string
  const changes: StringChange[] = []
  for (let i = 0; i < fromTuning.strings.length; i++) {
    if (fromTuning.strings[i] !== toTuning.strings[i]) {
      changes.push({
        stringNumber: i + 1,
        from: fromTuning.strings[i],
        to: toTuning.strings[i],
        semitones: calculateSemitones(
          fromTuning.strings[i],
          toTuning.strings[i]
        ),
      })
    }
  }

  // Determine effort level
  let effort: RetuningAnalysis['effort']
  let recommendation: string | undefined

  if (changes.length === 0) {
    effort = 'none'
  } else if (changes.length === 1) {
    effort = 'quick'
    recommendation = '~30 seconds between songs'
  } else if (changes.length <= 3) {
    effort = 'moderate'
    recommendation = '~2 minutes - plan transition time'
  } else {
    effort = 'full'
    recommendation = 'Consider guitar swap or setlist reorder'
  }

  return {
    stringsChanged: changes.length,
    effort,
    details: changes,
    recommendation,
  }
}

// Note to semitone mapping
const NOTE_TO_SEMITONE: Record<string, number> = {
  C: 0,
  'C#': 1,
  Db: 1,
  D: 2,
  'D#': 3,
  Eb: 3,
  E: 4,
  F: 5,
  'F#': 6,
  Gb: 6,
  G: 7,
  'G#': 8,
  Ab: 8,
  A: 9,
  'A#': 10,
  Bb: 10,
  B: 11,
}

function calculateSemitones(from: string, to: string): number {
  const fromSemitone = NOTE_TO_SEMITONE[from]
  const toSemitone = NOTE_TO_SEMITONE[to]
  let diff = toSemitone - fromSemitone

  // Handle octave wrapping (e.g., E to Eb could be -1 or +11)
  if (diff > 6) diff -= 12
  if (diff < -6) diff += 12

  return diff
}
```

## Migration Path

### Phase 1: Add JSONB Column (Non-Breaking)

```sql
-- Add new column, keep old one
ALTER TABLE songs
ADD COLUMN tuning JSONB;

-- Migrate existing data
UPDATE songs
SET tuning = jsonb_build_object(
  'name', guitar_tuning,
  'stringCount', 6,
  'strings', ARRAY['E', 'A', 'D', 'G', 'B', 'E']::text[],
  'isCustom', false
)
WHERE guitar_tuning IS NOT NULL;

-- Set default for songs without tuning
UPDATE songs
SET tuning = jsonb_build_object(
  'name', 'Standard',
  'stringCount', 6,
  'strings', ARRAY['E', 'A', 'D', 'G', 'B', 'E']::text[],
  'isCustom', false
)
WHERE tuning IS NULL;
```

### Phase 2: Update Application Code

1. Update TypeScript types to use `GuitarTuning` interface
2. Update SongRepository to handle JSONB serialization
3. Deploy tuning library and comparison algorithm
4. Add UI components for custom tunings
5. Add retuning indicators to setlist view

### Phase 3: Clean Up (After Verification)

```sql
-- Drop old column once migration verified
ALTER TABLE songs
DROP COLUMN guitar_tuning;
```
