## Confirmed bugs (documented — not fixed)

### BUG-1 — Duplicate free-text casts on one part · LOW/MEDIUM · data integrity

- **Repro:** host on an event → open a song → Assign a part → type a name → Add;
  reopen Assign → type the _same_ name → Add. Two rows persist; roster shows the name
  twice.
- **Root cause:** the unique index `uq_casting_slot_role_member (…, role_key, member_id)`
  dedups member casts (23505 → caught in `CastingAssignmentService.assign`), but
  free-text casts set `member_id = NULL` and Postgres treats NULLs as distinct, so the
  guard never fires.
- **Impact:** roster duplicates + orphan rows only. **Counts are unaffected** (the
  card/header/panel %s use per-(slot,role) distinct counting), so no numeric
  inconsistency.
- **Fix options (pick one):** (a) app-layer — before inserting a free-text cast, skip if
  a cast with the same `(slot, role, member_name)` exists; (b) DB — recreate the unique
  index with `NULLS NOT DISTINCT` **plus** a normalized-name guard (amend the local-only
  `20260703045901_casting.sql`; requires `db reset` + pgTAP + `lint:migrations`).

### BUG-2 — Cast counts stale on cold first paint · LOW · cosmetic · NOT reproducible

- On the very first navigation the header/pill showed a lower count that snapped correct
  on any interaction; every subsequent fresh reload rendered correctly. A first-load
  data-settling race, low confidence. Worth a glance at the `useCasting` initial-fetch
  vs `useEventDetail` ordering.

## Lower-confidence concerns (human review)

1. **e2e suite is not a trustworthy gate** — 28 red on a full parallel run, ~half flaky.
   Stabilize before relying on it: per-test Supabase isolation for multi-user specs,
   realtime-slot recreation, serialize signup-heavy specs.
2. **Test isolation on seed data** — the suite mutates the _seeded_ event `e1111111…`
   (adds participants). Multi-user specs shouldn't touch demo seed rows.
3. **Bidirectional friend requests** — `FriendService.insertRequest` only guards the
   `(me→target)` UNIQUE pair. If `target→me` is already pending, a `me→target` insert
   succeeds → two opposing pending rows. UI hides the Add button so it's hard to hit, but
   the service doesn't defend it. _Not live-reproduced._ Small guard if we want it.
4. **Hand accept → cast** double-assignment not exercised end-to-end (code path reviewed,
   looks sound).

  not coupled to `song_id` at the DB level; a lineup item can be `source='band'` with a
  null `song_id` (seed's "Mr. Brightside"). Intentional (resolve-later flow); add a CHECK
  only if we want to enforce it.
