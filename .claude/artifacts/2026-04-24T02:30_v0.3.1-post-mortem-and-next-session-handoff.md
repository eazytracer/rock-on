# v0.3.1 Post-Mortem & Next-Session Handoff

## What v0.3.1 actually closed

Five production bugs that surfaced within minutes of v0.3.0 deploy:

| #   | Symptom                                        | Root cause                                                                                  | Fix                                                                             |
| --- | ---------------------------------------------- | ------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------- |
| 1   | 403 on every jam REST call                     | Baseline `GRANT ON ALL TABLES TO authenticated` is a snapshot — didn't cover new jam tables | Migration `20260424000000` grants authenticated on jam tables                   |
| 2   | 404 on anonymous jam-view                      | No `GRANT ... TO service_role` anywhere (entire baseline gap)                               | Migration `20260424020000` grants all public tables + default privileges        |
| 3   | 401 on anonymous jam-view (before #2 surfaced) | Edge function deployed without `--no-verify-jwt`                                            | Redeploy; `FUNCTIONS.md` manifest + deploy script prevent recurrence            |
| 4   | `User abc123` / `Host` displayed               | Code read `user_profiles.display_name` only; rows never created by signup flow              | Fall back to `users.name` in `RemoteRepository.getJamParticipants` + `jam-view` |
| 5   | Share URL `?t=...` lost after remount          | Raw token only known at session creation; DB stores hash                                    | Persist to localStorage keyed by session id + graceful disable when unavailable |

Plus infrastructure meant to catch this class of bug before prod next time:

## What v0.3.1 missed

Caught in user testing post-deploy (by @eazytracer):

### 🚨 Anonymous jam-view UX is functionally useless

The anon view at `/jam/view/<code>?t=<token>` currently shows:

- Host display name ✓ (fixed in v0.3.1)
- **"Common songs" list** ✗ — this is intrinsically wrong for an anonymous viewer. "Common songs" means "songs shared by all active participants' personal catalogs." An anonymous viewer has no catalog, so they're neither a participant nor a contributor to the match set. Displaying common songs to them is meaningless at best, misleading at worst.
- **No setlist** ✗ — the anon page has no rendering for the host's broadcast setlist, and the `jam-view` edge function's payload doesn't even populate the `setlist` field. The schema scaffolding exists (`jam_sessions.settings.setlistItems` and a `setlist` field on `JamViewPublicPayload`) but the plumbing between the host's curated setlist and the edge function response was never wired up — the old "next-session handoff" doc called this out as Item 1 "In-session setlist builder + live broadcast" and that work was superseded by our unrelated hotfix sprint.

The product intent for anonymous viewing — confirmed by the user's expectation when testing — is **"a guest sees essentially the same view an authenticated participant sees: the curated setlist the band is working through."** What they currently see is irrelevant internal plumbing.

This is the biggest open item. See "Next session priorities" below.

### Hanging chads tracked but not fixed this cycle

These were flagged during the session and left for follow-up rather than piling more scope onto the hotfix:

1. **Raw view token only in localStorage.** Sharing a session URL from a device other than the one that created it produces a disabled Copy/QR button (per the graceful-disable fix). Real fix: store the raw token on `jam_sessions` in a host-only readable column, or surface it via a SECURITY DEFINER RPC gated by `host_user_id = auth.uid()`. Out of scope for v0.3.1; tracked here.
2. **Functionality catalog backfill.** Template and template-usage were established; Jam Sessions domain fully backfilled as exemplar. The remaining 10 domains (Auth, Bands, Songs, Setlists, Shows, Practices, Casting, Sync, Realtime, UI Layout) still use the pre-template simple listing format. Non-blocking; each will get the new format when its capabilities are next touched.
3. **E2E tests for v0.3.1 fixes never landed.** Playwright was blocked in the session where we'd have written them. Specifically missing:
   - `tests/e2e/jam/jam-share-url-persistence.spec.ts` — create session, refresh, verify share URL still has `?t=`
   - Expansion of `tests/e2e/jam/jam-view-anon.spec.ts` — full page render for anon user with real host name display (not just edge-function 200)
   - `tests/e2e/auth/post-signup-profile-state.spec.ts` — verify a freshly-signed-up user with no user_profiles row still has their name displayed in list views
4. **`jam-recompute` harness end-to-end verification.** I never ran the multi-user CLI harness against the post-hotfix state to confirm matches actually populate for two-user sessions. Likely works (the service_role grants opened the lock), but not explicitly verified. `npm run harness -- seed-songs alice && seed-songs bob && create-jam alice && join-jam bob && recompute` should demonstrate this end-to-end.
5. **Other surfaces reading `user_profiles.display_name` only.** Audited during v0.3.1; no additional fixes needed (BandMembersPage already has fallback, RealtimeManager reads `users.name`, AuthPages uses auth metadata not user_profiles). This was confirmed clean — noting here so the next session doesn't re-audit.
6. **`lint:migrations` strict mode.** Currently the linter allows the baseline migration's broad `GRANT ON ALL TABLES TO authenticated` pattern. If we ever split the baseline or adopt a stricter per-table policy, a `--strict` flag could require explicit per-table grants throughout. Low priority; document for future.
7. **jam-recompute auth mode.** Manifest says "verify JWT" (default). Never explicitly audited whether `jam-recompute` calls from the frontend include the JWT correctly. Smoke test passes (401 on anon GET), but haven't confirmed authenticated calls succeed. Likely fine since matches worked before the cascade.

## Next session priorities (suggested order)

### P0 — Rebuild anonymous jam-view around the actual product intent

The core UX work: decide what a guest SHOULD see and build that.

**Suggested shape:**

- Remove "Common Songs" from the anon payload. Guests don't benefit from it.
- Add the broadcast setlist as the primary content. Source: `jam_sessions.settings.setlistItems` — already an array of `{ id, displayTitle, displayArtist }` (so no ID resolution required for an anon viewer who has no catalog).
- Edge function updates:
  - Read `settings.setlistItems` from `jam_sessions`, project to `JamViewPublicPayload.setlist`
  - Keep host display name (already correct)
  - Drop or hide the matches field from the payload
  - Preserve the participant count (anonymous-safe signal that the session is populated)
- Frontend `JamViewPage` updates:
  - Render the setlist as the primary content
  - Add "live" realtime subscription so the setlist updates as the host reorders
  - Handle empty-setlist state clearly ("Host hasn't added any songs yet")
  - Remove the common-songs section

**Scaffolding notes from the April handoff:** the `JamSetlistItem` interface and `JamViewPublicPayload.setlist?` already exist in `src/models/JamSession.ts`. The work is wiring the edge function to populate it and wiring the page to render it. The earlier plan was Item 1 of the Phase B/C followup — that artifact was deleted when the feature was "shipped" but the setlist-broadcast piece genuinely wasn't.

### P1 — Add the missing E2E tests

All three noted above. Start with:

- `jam-view-anon.spec.ts` — exercises P0's new anon UX end-to-end
- `jam-share-url-persistence.spec.ts` — guards the v0.3.1 fix

### P2 — Functionality catalog backfill (incremental)

As each domain is touched for feature work, restructure its entries to the new template. Natural places to start: whichever domain the next feature cycle lands in.

### P3 — Permanent fix for share URL token

Option: add `view_token_raw` column to `jam_sessions` with a host-only SELECT policy:

```sql
ALTER TABLE public.jam_sessions
  ADD COLUMN view_token_raw TEXT;
-- Set only at create time; never exposed via PostgREST to non-hosts
CREATE POLICY "jam_sessions_select_view_token_raw_host_only" ...
```

Or keep localStorage + accept the limitation. User feedback will tell us if this matters in practice; for now most users create and share from the same session, so localStorage is fine.

## What went well

- The red-→-green linter demo was convincing ("here's how we'd catch this next time") and landed as CI-ready tooling
- The catalog template shift from "one-liner entries with test paths" to "Who-can / Who-cannot / positive + negative tests" captures the right level of coverage thinking
- Honest course-correction mid-cycle (stop applying fixes to prod first) produced a better hotfix PR

## What went poorly

- Initial fixes were applied to prod before code review. Called out and corrected; documented in `pre-deploy-checklist.md` as explicit policy
- The anon-view UX was never reviewed before v0.3.0 shipped. The edge function "returned 200" but the product it enabled was incomplete. "Status 200" ≠ "feature done"
- No staging environment — every deploy is straight to prod. Adding one is probably the single highest-leverage process improvement, but out of scope here
