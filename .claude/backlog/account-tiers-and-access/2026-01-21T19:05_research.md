# Research: Account Tiers and Access Control System

## Feature Request Summary

Implement a tiered account system with usage limits for free accounts and unlimited access for pro accounts, with ability to distribute access codes to friends for testing. The system should:

- Allow anyone to create a free account
- Enforce usage limits on free tier (e.g., 30 songs, 8 band members)
- Support access codes/QR codes to grant pro tier access
- Consider personal song catalog feature before rollout
- Be flexible for future expansion and payment integration

## Account Ownership Model Analysis

### Critical Question: Should accounts be tied to people or bands?

**Current State:**

- Accounts ARE tied to people (users table)
- Bands are entities that users belong to via `band_memberships`
- One user can belong to multiple bands
- Each band has an owner (created_by)

### Option A: User-Based Tiers (RECOMMENDED)

**How it works:**

- Each user has an account tier (free/pro)
- Limits apply to resources the user OWNS across all bands
- Example: User can create max 30 songs total (across personal + bands they own)
- Band member limits apply per band the user owns

**Pros:**

- ✅ Simpler model: One tier per user
- ✅ Clear ownership: User pays, user gets benefits
- ✅ Prevents abuse: Can't create unlimited bands to bypass limits
- ✅ Matches payment model: Individual subscribes, not band
- ✅ Personal song catalog fits naturally

**Cons:**

- ❌ Band collaborators don't benefit from owner's pro tier
- ❌ More complex limit checking (must aggregate across bands)
- ❌ User might hit limits if active in multiple bands

### Option B: Band-Based Tiers

**How it works:**

- Each band has a tier (free/pro)
- Limits apply to the band regardless of members
- Owner pays to upgrade band
- All band members benefit from pro features

**Pros:**

- ✅ Simpler limit checking (per band)
- ✅ All collaborators benefit from pro tier
- ✅ Matches collaborative nature of music

**Cons:**

- ❌ User can create unlimited free bands to bypass limits
- ❌ Personal song catalog doesn't fit (needs user-level tier)
- ❌ Complex ownership transfers
- ❌ Harder to monetize (bands share one subscription)

### Option C: Hybrid Model

**How it works:**

- Users have account tiers for personal features
- Bands have separate tiers for band features
- Two parallel limit systems

**Pros:**

- ✅ Flexibility for both individual and band use cases
- ✅ Can monetize both levels

**Cons:**

- ❌ Most complex to implement
- ❌ Confusing for users (two separate tier systems)
- ❌ Higher development and maintenance cost

### Recommendation: User-Based Tiers (Option A)

**Rationale:**

1. **Personal Song Catalog:** User must have personal storage limits separate from bands they collaborate with
2. **Clear Ownership:** User creates account, user controls tier
3. **Abuse Prevention:** Can't bypass limits by creating multiple bands
4. **Future Payment:** Individual subscriptions are easier than band subscriptions
5. **Flexibility:** User can be in multiple bands without hitting limits (if only contributing, not owning)

## Access Code System Design

### Requirements

- Generate unique codes
- Validate codes
- Assign pro tier to user
- Track code usage
- Support QR codes

### Code Format

```typescript
// Format: ROCK-YYYY-XXXXXX (e.g., ROCK-2026-4B3D9F)
function generateAccessCode(): string {
  const year = new Date().getFullYear()
  const random = crypto.randomUUID().slice(0, 6).toUpperCase()
  return `ROCK-${year}-${random}`
}
```

### Access Code Flow

1. Admin creates code in admin panel
2. Code saved to `access_codes` table
3. QR code generated client-side (using qrcode.react)
4. User scans QR or enters code manually
5. Code validated (active, not expired, under limit)
6. User's `accountTier` updated to 'pro'
7. Redemption recorded in `code_redemptions` table

### Database Schema

**Add to users table:**

```sql
ALTER TABLE users ADD COLUMN account_tier TEXT DEFAULT 'free'
  CHECK (account_tier IN ('free', 'pro'));
ALTER TABLE users ADD COLUMN tier_updated_at TIMESTAMPTZ DEFAULT now();
ALTER TABLE users ADD COLUMN tier_updated_by UUID REFERENCES users(id);
```

**New table: access_codes**

```sql
CREATE TABLE access_codes (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  code TEXT UNIQUE NOT NULL,
  tier TEXT NOT NULL CHECK (tier IN ('pro')),
  created_by UUID REFERENCES users(id) NOT NULL,
  created_date TIMESTAMPTZ NOT NULL DEFAULT now(),
  expires_at TIMESTAMPTZ,
  max_redemptions INTEGER NOT NULL DEFAULT -1,
  redemptions INTEGER NOT NULL DEFAULT 0,
  active BOOLEAN NOT NULL DEFAULT true,
  notes TEXT,
  version INTEGER NOT NULL DEFAULT 1,
  last_modified_by UUID REFERENCES users(id)
);
```

**New table: code_redemptions**

```sql
CREATE TABLE code_redemptions (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  code_id UUID REFERENCES access_codes(id) ON DELETE CASCADE NOT NULL,
  user_id UUID REFERENCES users(id) ON DELETE CASCADE NOT NULL,
  redeemed_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  ip_address INET,
  UNIQUE(code_id, user_id)  -- User can only redeem each code once
);
```

## Risk Analysis

### High Risk

| Risk                               | Impact                              | Mitigation                                                            |
| ---------------------------------- | ----------------------------------- | --------------------------------------------------------------------- |
| Breaking Changes to Existing Users | Existing users >30 songs blocked    | Grandfather existing users as 'pro', apply limits to new signups only |
| Complex Limit Checking Logic       | Performance issues with aggregation | Cache limit counts, use proper indexes, denormalize if needed         |
| Access Code Abuse                  | Codes shared publicly               | Set redemption limits, track IPs, expire codes, rate limit validation |

### Medium Risk

| Risk                          | Impact                                  | Mitigation                                |
| ----------------------------- | --------------------------------------- | ----------------------------------------- |
| Personal Song Catalog Missing | User-based tiers don't make sense       | Implement personal catalog FIRST          |
| User Confusion (Two Contexts) | Users don't understand personal vs band | Clear UI distinction, onboarding tutorial |
| Migration Path Unclear        | Existing users confused                 | Email notification, in-app banner, FAQ    |

### Low Risk

| Risk                    | Impact                  | Mitigation                                            |
| ----------------------- | ----------------------- | ----------------------------------------------------- |
| QR Code Library Issues  | QR codes don't generate | Use well-maintained library, fallback to manual entry |
| Database Storage Growth | Limits too high         | Monitor usage, adjust limits based on data            |
